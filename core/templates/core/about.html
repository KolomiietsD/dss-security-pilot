Про систему

Information Security DSS — навчальний прототип системи підтримки рішень (Decision Support System) для SOC / інформаційної безпеки.

Що це за проєкт?

Система збирає дані з декількох джерел (Wazuh та CrowdStrike), уніфікує їх у спільну структуру та відображає на дашбордах. Далі поверх цих даних запускається інтелектуальний модуль, який: групує події у часові епізоди, оцінює ризик і формує пріоритет хостів.

Важливо: система не замінює аналітика. Її задача — зменшити шум, прискорити триаж (triage) та допомогти швидко знайти критичні вузли/епізоди, які потребують уваги.

Як працює система (пайплайн)
1) Збір та уніфікація даних
  • Wazuh: агенти, alerts, SIEM events (логи з хостів)
  • CrowdStrike: хости, детекти (endpoint security)
  • Нормалізація ключів: hostname/IP, приведення полів до єдиного формату
  • Формування списку assets (активів) і списку events (подій)
2) Епізоди та аналітика
  • Групування подій у епізоди по часовому вікну (≈ 90 секунд)
  • Розрахунок ознакового вектора епізоду (кількість подій, severity, різноманіття, джерела тощо)
  • Кластеризація K-Means та розрахунок ML risk_score
  • Семантична оцінка епізоду/подій модулем BERT/NLP
  • Формування пріоритету хоста модулем MLP (багатошаровий персептрон)
Результат: система показує “Top host risks”, список епізодів за ризиком, події з деталями та підказує, які вузли найімовірніше потребують реагування.
Джерело даних
CrowdStrike

Використовується для отримання інформації про endpoint-и (хости) та детекції (detections). Це дає “сигнали” від EDR/XDR, які часто мають високий контекст інциденту.

  • host inventory (hostname, IP, статус)
  • detections (події безпеки, поведінка, IOC)
Перейти до хостів
Джерело даних
Wazuh (SIEM)

Wazuh забезпечує збір логів з хостів, подій безпеки та стану агентів. Дає “низькорівневий” контекст: авторизації, процеси, системні події, правила, рівні тощо.

  • agents (статус, ОС, last_keepalive)
  • alerts / siem events (правила, level, опис, групи)
Перейти до Wazuh
Шар інтеграції
Уніфікація подій

Дані з різних джерел зводяться у спільну структуру. Це потрібно, щоб: (1) можна було порівнювати події, (2) групувати їх у епізоди, (3) будувати ML-ознаки.

  • нормалізація hostname / IP
  • єдині поля: time, severity, user, process, description
  • позначення джерела: Wazuh / CrowdStrike
ML-модуль
Кластеризація K-Means

Епізоди подій перетворюються в числові вектори та кластеризуються методом K-Means. Це допомагає виділити типові сценарії активності (наприклад: “успішні логіни”, “серії помилок”, “детекти EDR”).

  • ознаки епізоду: кількість подій, max severity, унікальні користувачі/процеси
  • вихід: cluster_id + внесок у risk_score
NLP-модуль
BERT-аналіз тексту подій

Для семантичної інтерпретації описів подій використовується модуль BERT/NLP. Він додає контекст: що саме означає подія “людською мовою”, та чи має вона ознаки загрози.

  • вхід: текстові описи / message / rule.description
  • вихід: bert_label + bert_score
ML-модуль
MLP (багатошаровий персептрон)

Модуль пріоритезації хостів агрегує результати епізодів і формує підсумковий пріоритет активу. На вході — фічі по хосту (кількість епізодів, ризики, кластери, BERT-мітки), на виході — score/level.

  • Архітектура: Input → Dense → ReLU → Dense → Sigmoid/Softmax
  • Результат: ai_priority_score (0..1) + ai_priority_level
Фокус
Підтримка рішень

Система допомагає в типовому циклі SOC: збір → кореляція → триаж → пріоритезація. Вона показує “де гаряче” і зменшує навантаження на ручний перегляд всіх логів.

  • Top-5 хостів за AI-пріоритетом
  • таблиця епізодів за ризиком
  • деталі подій + контекст
Мета та обмеження прототипу

Проєкт створюється як навчальна/дослідницька робота у сфері інформаційної безпеки та data science. Він демонструє підхід до інтеграції SIEM/EDR даних та побудови інтелектуальної надбудови для оцінки ризику.

Сильні сторони
  • інтеграція декількох джерел (Wazuh + CrowdStrike)
  • уніфікація подій і кореляція через епізоди
  • ML/NLP компоненти (K-Means, BERT, MLP)
  • візуалізація для SOC-триажу
Поточні обмеження
  • якість оцінки залежить від кількості/різноманіття подій
  • для стабільності ML потрібне накопичення історичних даних
  • частина скорингу базується на евристиках (особливо на ранньому етапі)
  • прототип орієнтований на демонстрацію підходу, а не production-SOC

Подальші кроки: підвищення стабільності моделей, накопичення датасету, покращення UX, оптимізація запитів до API та розширення сценаріїв (MITRE-орієнтована кореляція, правила реагування).
